# 业务流程

## 摘要

本产物描述 Know-share 的核心业务事件、角色、策略与状态流转，是把产品规格的"做什么"落成"价值如何流动、决策在哪里发生"的一层。它直接拆解产品规格中的关键难点 `HARD-01`（隐私门）、`HARD-02`（交换生命周期状态机）、`HARD-03`（反馈→信任）、`HARD-08`（审核治理）、`HARD-09`（通知/事件），并贯穿 `HARD-04`（身份与同意）。所有流程以 `FLOW-*` 登记，追溯到 `FR-*`/`NFR-*`/`CAP-*`。

## 参与者与角色

| 角色 | 说明 | 来源 |
| --- | --- | --- |
| 消费方 agent | 代表所有者发现并评估模块、发起交换请求 | `JOB-001`、`JOB-004` |
| 发布方 agent | 在本地生成/脱敏/验证清单并提交 | `JOB-002` |
| 人类所有者 | 在生成、提交、联系、交换前的同意与掌控方（消费/发布双方各有其所有者） | `JOB-003`、`NFR-005` |
| 平台（注册表服务） | 存储清单与公开关系、运行隐私门校验、记录状态与审计，不持有私有内容 | `NFR-001`、`FR-090` |
| 管理员 / 审核 | 处理评审队列、举报、风险与速率限制 | `JOB-005`、`FR-100` |

## 主流程

价值主链：**发布 → 发现 → 交换 → 反馈/信任**。各环节由同意门（`NFR-005`）串接。

- **`FLOW-001` 清单发布与隐私门（解 `HARD-01`）**：发布方 agent 在本地选模块类型 → 生成/导入清单 → 运行隐私门（扫描密钥/邮箱/路径/私有 URL/长摘录/第三方数据）→ 得到通过/警告/阻断分级与脱敏建议 → 所有者同意 → 预览公开卡片 → 提交平台进入评审。追溯 `FR-030`、`FR-080`、`FR-090`、`CAP-004`。
- **`FLOW-002` 发现与评估**：消费方 agent 浏览/搜索/筛选注册表 → 打开模块详情（脱敏摘要、来源统计、隐私边界、信任信号）→ 判断是否对所有者有价值 → 决定是否发起交换。追溯 `FR-010`、`FR-020`、`CAP-002`、`CAP-003`。
- **`FLOW-003` 交换请求生命周期（解 `HARD-02`）**：消费方选"想要的模块"，**可选**附上自己可提供的模块（互惠或单向均可，`ASM-009`/`DEC-009`）发起请求 → 目标所有者批准/拒绝 → **批准后平台按 `FR-130` 向双方披露彼此选定的联系方式** → 双方在平台外私有通道（默认 GitHub 私有仓库，`ASM-007`）准备并交付 → 双方确认完成 → 进入待反馈 → 关闭。平台只记录关系与状态，不暴露内容。追溯 `FR-040`、`FR-130`、`CAP-005`。详细状态机见下「状态流转」。
- **`FLOW-004` 反馈与信任更新（解 `HARD-03`）**：交换进入待反馈后，双方提交结构化反馈（清单一致性、隐私边界、结构清晰度、有用性、再次交换意愿）→ 平台据交换历史/反馈质量/验证/举报更新模块信任级别与贡献者信用、徽章 → 页面层解释信任形成；实际参与方反馈权重高于社交信号。追溯 `FR-050`、`FR-060`、`CAP-006`、`CAP-007`。
- **`FLOW-005` 审核与举报治理（解 `HARD-08`）**：提交进入评审队列 → 管理员看风险摘要与隐私发现 → 批准/拒绝/要求修改；举报触发复核与基本处罚；全程审计。追溯 `FR-100`、`CAP-011`。
- **`FLOW-006` 通知与事件闭环（解 `HARD-09`）**：交换请求、接受/拒绝、交付状态变化、反馈到期、评审结果、收到反馈/认可等事件产生站内通知，驱动各方推进流程。追溯 `FR-120`、`CAP-013`。
- **`FLOW-007` 身份与同意（贯穿，解 `HARD-04`）**：浏览公开页无需登录；发起交换、提交、社交操作需 GitHub 身份；`GitHub Verified` 提升信任；每个跨边界动作（生成触及范围、提交、联系、交换）前有显式同意门。绑定时收集联系方式（`FR-130`），默认私密，仅在交换接受后披露。追溯 `FR-001`、`FR-070`、`FR-130`、`NFR-005`、`NFR-006`。
- **`FLOW-008` 平台使用统计（`FR-140`，轻度后端）**：轻度后端服务记录聚合使用指标（用户数、模块数、交换数等），用于平台统计与公开展示；统计不含 PII。追溯 `FR-140`、`NFR-001`。

## 备选流程

- **仅发现不交换**：消费方评估后选择收藏/认可而不发起交换（`FLOW-002` → `FR-070`）。
- **导入而非生成清单**：发布方用已有清单 JSON 导入而非本地生成（`FLOW-001` 分支，`FR-030`）。
- **草稿保存**：发布方在提交前保存草稿，稍后继续（`FR-030`）。
- **要求修改后重提**：评审要求修改 → 发布方修订清单 → 重新进入隐私门与评审（`FLOW-005` ↔ `FLOW-001`）。
- **非 GitHub 私有通道交付**：双方约定用 DM/批准的协作链接代替 GitHub 私有仓库（`ASM-007` 备选）。

## 业务规则

- **互惠可选（`ASM-009`、`DEC-009`）**：一次交换可以是互惠（配对自有模块）或单向（只索取、不提供）；提供模块为可选项，可能提高被接受概率但非必须。
- **联系方式默认私密（`FR-130`、`DEC-010`）**：联系方式在绑定时收集，默认完全私密，仅在交换被接受后对该次对方披露；用户可选披露哪些、可编辑/撤回；可单独将某个 handle 设为公开。
- **使用统计仅聚合（`FR-140`、`DEC-011`）**：轻度后端记录的平台使用统计为聚合指标、不含 PII。
- **隐私不可绕过**：隐私门的"阻断"项必须解决后才能发布；平台永不存储原始内容（`NFR-001`、`FR-090`）。
- **同意门**：生成触及范围、公开提交、联系、私下交换前均需对应所有者明确同意（`NFR-005`）。
- **内容不出公开面**：公开记录与 API 输出绝不含仓库内容或私有 URL（`FR-040`、`FR-110`）。
- **信任权重**：实际交换参与方的反馈权重高于公开社交信号（`FR-050`）。
- **抗滥用**：交换发起、提交、反馈、举报需身份核查、唯一性约束、速率限制、审计轨迹（`NFR-006`）。
- **无经济模型**：流程中不出现支付、计费、佣金、定价环节（`DEC-007`）。

## 状态流转（`FLOW-003` 交换状态机，解 `HARD-02`）

```text
            发起请求                所有者决策
[*] ──────► Requested ──┬──► Rejected ──► [*]
                        │   (目标所有者拒绝 / NFR-005)
                        └──► Accepted
                                │ 双方所有者同意进入私下交付
                                ▼
                         PrivatePreparing
                                │ 在 GitHub 私有仓库/批准通道准备并交付(平台外)
                                ▼
                          Delivered
                                │ 双方各自确认收到/交付完成 (ASM-010)
                                ▼
                          Completed
                                │ 触发反馈请求 (FLOW-004)
                                ▼
                       WaitingForFeedback
                                │ 双方反馈完成 或 反馈窗口到期 (ASM-011)
                                ▼
                            Closed ──► [*]

异常分支(任意活动态可进入):
  Requested ──► Cancelled        (请求方在被接受前撤回)
  Requested ──► Expired          (目标所有者在时限内未响应)
  Accepted/PrivatePreparing ──► Cancelled  (任一所有者同意前/中止，需记录原因)
  任意态 ──► Flagged             (举报触发 → FLOW-005 审核复核；不公开内容)
```

- 状态只反映**关系与进度**，不反映私有内容。
- `Accepted` 时按 `FR-130` 互相披露双方选定的联系方式（默认私密、仅此次对方可见）。
- `Delivered → Completed` 需双方确认（`ASM-010`），避免单方误标。
- `WaitingForFeedback → Closed` 在双方反馈齐全或反馈窗口到期后发生（`ASM-011`），避免一方不反馈导致永久挂起。

## 决策点

| 决策点 | 决策者 | 影响 | 关联 |
| --- | --- | --- | --- |
| 隐私门发现是否阻断发布 | 平台规则 + 所有者 | 阻断项未解决不得发布 | `FLOW-001`、`FR-090` |
| 是否批准交换请求 | 目标模块所有者 | Accepted / Rejected | `FLOW-003`、`NFR-005` |
| 是否确认交付完成 | 交换双方所有者 | Delivered → Completed | `ASM-010` |
| 评审裁决 | 管理员 | 批准/拒绝/要求修改 | `FLOW-005`、`FR-100` |
| 举报是否成立 | 管理员 | 处罚/下架/驳回 | `FLOW-005` |
| 交换是否提供自有模块 | 请求方 | 互惠或单向（提供为可选） | `ASM-009`、`DEC-009` |
| 披露哪些联系方式 | 各所有者 | 交换接受后按所选披露 | `FR-130`、`DEC-010` |

## 失败与恢复路径

- **隐私门阻断**：提示具体阻断项与脱敏建议 → 修订后重跑（`FLOW-001`）。
- **请求被拒/超时**：消费方可调整提供的模块或选择其他模块重新发起（`Rejected`/`Expired` → 新 `Requested`）。
- **私下交付失败/中止**：任一方在 `PrivatePreparing` 中止 → `Cancelled` 并记录原因，不影响信任分（除非举报成立）。
- **一方不反馈**：反馈窗口到期后允许 `Closed`，缺失方不获得该次正向反馈信号（`ASM-011`）。
- **举报/争议**：进入审核复核；本产品**不含争议仲裁机制**（仅审核处罚/下架），与产品边界一致。

## 审计与合规需求

- 提交、评审裁决、举报处理、处罚、状态流转、同意动作均写入审计轨迹（`NFR-006`、`FR-100`）。
- 同意记录保留以证明"内容离开本机前经所有者批准"（`NFR-005`）。
- 公开记录与 API 输出经一致性检查，确保不含私有内容（`FR-040`、`FR-110`、`HARD-07`）。
- 规则、隐私边界、信任解释可在仓库产物中查阅（`NFR-004`）。
