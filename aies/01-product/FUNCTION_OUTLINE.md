## 产品任务

| Job ID | 主要参与者 | 任务 | 来源 |
| --- | --- | --- | --- |
| `JOB-001` | 消费方 agent | 发现可能对其所有者有用的公开知识模块清单（manifest）。 | README、发现页 UI |
| `JOB-002` | 发布方 agent | 生成、脱敏处理、验证并提交一份隐私安全的清单。 | Agent Skills UI、Submit Module UI |
| `JOB-003` | 人类所有者 | 在公开提交、联系和私下交换之前保持掌控。 | 隐私模型 |
| `JOB-004` | 交换参与方 | 请求、批准、完成并评价一次私下的知识交换。 | Exchange Records UI |
| `JOB-005` | 社区 / 未来管理员 | 通过验证、反馈、审核和透明记录来建立信任。 | Trust Profile UI、管理员概览 |

## 已考虑的分解方式

| 方式 | 描述 | 权衡 | 建议 |
| --- | --- | --- | --- |
| `APPROACH-A` 页面优先 | 每个页面分配一个 agent：发现、详情、交换、个人资料、技能、提交。 | UI 进展快，但共享 schema 与信任逻辑可能漂移。 | 在数据契约存在之后有用，而非最先采用。 |
| `APPROACH-B` 领域优先 | 每个领域分配一个 agent：注册表、交换、信任、隐私、agent-kit、审核。 | 模型边界更强，但更晚才能看到完整页面。 | 最适合后端与契约阶段。 |
| `APPROACH-C` 垂直切片混合 | 先稳定共享契约，再按发布关键的垂直旅程拆分。 | 前期规划略多，对并行 agent 的返工最少。 | 推荐。 |

推荐路径：采用 `APPROACH-C`。先定义共享实体、权限边界、清单 schema、信任事件和交换生命周期。然后并行推进诸如发现、模块详情、提交、交换记录、信任档案、Agent 技能和管理员审核等垂直切片。

## 能力图谱

### CAP-001 公共站点外壳与身份

- 用户价值：用户和 agent 可以在公共注册表、交换记录、信任网络、Agent 技能、开源仓库和提交流程之间定位自己。
- 输入：公共导航标签、GitHub 登录状态、全局搜索查询、当前路由、已登录时的 GitHub 身份。
- 输出：一致的导航、已认证/匿名 UI 状态、全局搜索入口、提交模块入口。
- 依赖：`CAP-002`、`CAP-003`、未来的认证决策。
- 验收说明：
  - `FR-001` 全局导航暴露发现、交换记录、信任网络、Agent 技能以及仓库/帮助界面。
  - 包含 GitHub 登录状态，但浏览公开页面无需登录。
  - 包含全局搜索，在相应规格存在后路由到模块、主题、用户或交换搜索。
  - 包含一个提交模块入口，可在实际提交前要求 GitHub 身份。
- 推迟项：完整的账户设置、通知中心和邮件偏好。

### CAP-002 发现 / 注册表

- 用户价值：消费方 agent 或人类可以快速判断某个知识模块是否值得查看或请求。
- 输入：公开清单元数据、模块类型、主题、语言、信任级别、新鲜度、交换次数、收藏、反馈数量。
- 输出：可搜索可筛选的模块列表、模块卡片、主题标签、公共平台统计、请求交换 CTA。
- 依赖：`CAP-003`、`CAP-006`、`CAP-010`。
- 验收说明：
  - `FR-010` 展示模块卡片，含类型、标题、脱敏摘要、标签、所有者句柄、信任级别、交换次数、收藏数和请求操作。
  - 包含按标题、摘要、主题、作者和关键词搜索。
  - 包含按模块类型、主题、信任级别、语言和新鲜度筛选；支持按最新、交换次数、反馈质量或收藏排序。
  - 包含热门主题和平台统计，且不暴露私有内容。
  - 为已登录用户包含收藏或已保存模块。
  - 包含从模块卡片发起的请求交换入口。
- 推迟项：个性化推荐、合集、排序算法和 agent 侧打分 UI。

### CAP-003 知识模块详情

- 用户价值：消费方 agent 可以在提议交换之前查看模块的公开证据和隐私边界。
- 输入：模块清单、来源统计、覆盖的问题、所有者档案摘要、信任指标、反馈、隐私声明、内容承诺。
- 输出：模块详情页、清单预览/完整视图、来源统计、隐私边界、交换请求面板、举报入口。
- 依赖：`CAP-002`、`CAP-006`、`CAP-008`、`CAP-010`。
- 验收说明：
  - `FR-020` 展示模块身份：类型、标题、标签、所有者、信任级别、已完成交换、已验证反馈、收藏数和更新日期。
  - 包含脱敏摘要和覆盖的问题。
  - 包含来源统计，例如文件数、图表、代码片段、估算字数和内容类型分布。
  - 包含清单预览和完整清单视图。
  - 包含隐私边界和内容承诺。
  - 包含精选的交换后反馈。
  - 包含请求交换和举报模块操作。
- 推迟项：版本历史、清单 diff、公开评论串和 agent 生成的价值分析。

### CAP-004 清单提交与隐私门

- 用户价值：发布方 agent 可以提交一份有用的清单，同时人类所有者保持受保护和掌控。
- 输入：本地 agent 输出、清单 JSON、模块类型、脱敏处理报告、扫描发现、用户确认、草稿状态。
- 输出：提交向导、隐私扫描结果、脱敏处理建议、公开卡片预览、草稿/提交状态。
- 依赖：`CAP-009`、`CAP-010`、未来的评审工作流。
- 验收说明：
  - `FR-030` 提供多步提交流程：选择模块类型、生成/导入清单、运行隐私门、预览公开卡片、提交平台、准备私下交换包。
  - 包含通过、警告和阻断三类隐私发现。
  - 包含针对人物、公司、项目名、路径和代码片段的泛化建议。
  - 包含在上传前下载或保存清单。
  - 包含提交前的公开卡片预览。
  - 包含保存草稿和提交以供评审。
- 推迟项：富在线清单编辑器、批量导入、多语言变体和自动 GitHub PR 提交。

### CAP-005 交换请求与公开交换记录

- 用户价值：平台让交换关系和结果透明，同时不暴露私有仓库内容。
- 输入：请求用户、目标模块、提供的模块、接受/拒绝、交换状态、私下交付状态、验证结果、反馈。
- 输出：交换请求流程、公开交换台账、交换详情页、私有仓库交付清单、状态徽标。
- 依赖：`CAP-002`、`CAP-003`、`CAP-006`、`CAP-010`、未来的服务契约。
- 验收说明：
  - `FR-040` 通过选择一个要请求的模块和一个自有的待提供模块来发起交换。
  - 包含生命周期跟踪：requested、accepted、private repo preparing、completed、waiting for feedback、closed、rejected。
  - 包含公开记录，含参与方、GitHub 验证状态、已交换模块、完成时间、验证结果和反馈摘要。
  - 包含私下交付跟踪，但不公开泄露仓库内容或私有 URL。
  - 包含带有 agent 验证摘要和参与方反馈的交换详情。
  - 包含提醒或待反馈状态。
- 推迟项：平台托管的 GitHub App 自动化、争议解决、多方交换和付费/咨询型交换形式。

### CAP-006 反馈与信任系统

- 用户价值：用户可以通过结构化反馈和透明的信任信号来判断模块和贡献者的质量。
- 输入：已完成交换、参与方反馈、验证结果、收藏、举报、GitHub 验证、历史行为。
- 输出：模块信任级别、用户信用分、徽章、反馈质量维度、信任计算说明。
- 依赖：`CAP-005`、`CAP-007`、`CAP-008`、`CAP-010`。
- 验收说明：
  - `FR-050` 在交换后收集结构化反馈：清单一致性、隐私边界、结构清晰度、有用性、再次交换意愿、公开文本。
  - 包含从交换历史、反馈质量、验证和举报得出的可见信任级别。
  - 包含徽章，例如高质量贡献者、可信交换伙伴、优质反馈者、活跃交换者和隐私守护者。
  - 将实际交换参与方的反馈权重设得高于公开社交信号。
  - 在页面层面解释信任分的形成。
- 推迟项：反欺诈打分、加权图信任、异常检测和公开信任网络可视化。

### CAP-007 用户信任档案

- 用户价值：用户或 agent 可以评估贡献者的公开声誉和模块组合。
- 输入：GitHub 档案、已验证状态、已发布模块、已完成交换、反馈、徽章、信用趋势、兴趣。
- 输出：公开用户档案、指标、已发布模块列表、交换历史、认可墙、反馈质量细分。
- 依赖：`CAP-002`、`CAP-005`、`CAP-006`。
- 验收说明：
  - `FR-060` 展示 GitHub 身份、GitHub Verified 标记、档案摘要、领域标签、加入日期和近期活动。
  - 包含信任级别、信用分、分数趋势、已完成交换次数、已验证反馈数、平均反馈质量和再次交换意愿率。
  - 包含已发布模块及其交换/收藏/反馈数量。
  - 包含公开交换历史和获得的认可。
  - 包含反馈质量维度。
- 推迟项：关注用户、自定义档案主题、屏蔽名单和推荐图。

### CAP-008 社交信号与社区操作

- 用户价值：轻量级操作帮助用户保存、认可和讨论模块，同时不压过已验证的交换反馈。
- 输入：已登录用户、模块、档案、交换记录、评论文本、收藏/点赞操作、举报。
- 输出：收藏、点赞/认可、评论、举报、审计轨迹。
- 依赖：`CAP-002`、`CAP-003`、`CAP-006`、未来的认证。
- 验收说明：
  - `FR-070` 允许每位已登录用户对每个模块进行一次有效收藏。
  - 包含认可/点赞操作，其信任权重低于交换后反馈。
  - 在页面规格包含的地方支持基本评论或反馈片段。
  - 保持社交操作可审计且受速率限制。
- 推迟项：嵌套回复、超出管理员基础范围的评论审核 UI、公开/私有收藏夹。

### CAP-009 Agent 技能 / MCP 工具

- 用户价值：真正涉及隐私敏感的工作发生在用户的本地 agent 环境中。
- 输入：本地知识源路径、原始笔记/文档、清单 JSON、交换 ID、模块 ID、私有仓库/项目路径。
- 输出：清单、脱敏处理摘要、验证报告、私有仓库包、提交的反馈载荷、安装/配置示例。
- 依赖：`CAP-004`、`CAP-005`、`CAP-010`、`CAP-012`。
- 验收说明：
  - `FR-080` 展示 Agent 技能目录，含 Create Manifest、Redact Knowledge、Validate Manifest、Package Private Repo、Submit Feedback。
  - 为每个技能包含安装/文档入口。
  - 包含 MCP 配置示例和本地命令示例。
  - 支持来源类别：Obsidian、Logseq、Notion、Markdown、Yuque、Lark Docs、本地文件夹、自定义格式。
  - 为每个技能标注隐私级别和本地执行状态。
- 推迟项：真实 MCP 服务器实现、本地模型脱敏处理、自动 GitHub 仓库创建和签名的 agent 身份。

### CAP-010 隐私、安全与同意

- 用户价值：只有当产品让私有数据边界明确且可强制执行时，它才值得信任。
- 输入：清单内容、脱敏处理发现、敏感度声明、联系方式、用户同意、举报。
- 输出：隐私检查清单、同意节点、扫描警告/阻断、内容承诺、下架/举报状态。
- 依赖：所有产生数据的能力。
- 验收说明：
  - `FR-090` 公共平台存储清单和公开关系，而非原始笔记或私有仓库内容。
  - 在清单生成触及某个范围之前、公开提交之前、联系之前和私下交换之前，均要求同意。
  - 扫描密钥、邮箱、电话号码、ID、私有路径、私有 URL、长摘录和敏感的第三方数据。
  - 要求敏感度声明和脱敏处理说明。
  - 展示内容承诺以保证清单/包的一致性，且不暴露内容。
  - 支持举报和下架路径。
- 推迟项：强 PII 检测、Merkle 证明实现、本地 LLM 脱敏处理和特定策略的合规模块。

### CAP-011 管理与审核

- 用户价值：在信任能够规模化之前，公开质量和隐私安全需要一个运营闭环。
- 输入：已提交清单、隐私扫描结果、举报、风险级别、提交者信号、速率限制状态、审计日志。
- 输出：评审队列、风险摘要、批准/拒绝/要求修改操作、举报处理、审计轨迹。
- 依赖：`CAP-004`、`CAP-006`、`CAP-008`、`CAP-010`。
- 验收说明：
  - `FR-100` 按待处理、被标记、举报、速率限制和审计日志展示审核队列。
  - 包含清单风险级别、隐私风险摘要、问题列表和提交者信号。
  - 允许批准/发布、拒绝和要求修改。
  - 跟踪管理员操作历史。
  - 支持举报处理和基本的用户/模块处罚。
- 推迟项：自动化策略引擎、批量审核、申诉工作流和高级滥用检测。

### CAP-012 开放 API 与 agent 可读注册表

- 用户价值：agent 无需抓取页面即可与 Know-share 集成。
- 输入：公共注册表数据，模块/用户/交换/反馈筛选条件，已认证的提交或反馈载荷。
- 输出：公开模块/用户/交换 API、提交 API、反馈 API、搜索 API、速率限制响应。
- 依赖：`CAP-002`、`CAP-003`、`CAP-005`、`CAP-006`、`CAP-009`、`CAP-010`。
- 验收说明：
  - `FR-110` 为公开模块、用户、交换记录和搜索提供读取 API。
  - 在契约获批后提供已认证的提交和反馈操作。
  - 确保 API 输出绝不包含私有内容。
  - 应用速率限制和滥用控制。
- 推迟项：GraphQL、webhook、公共注册表镜像和签名的 agent 请求。

### CAP-013 通知

- 用户价值：交换和评审工作流需要提醒来闭环。
- 输入：交换请求、接受/拒绝、评审决定、反馈到期、评论/认可事件。
- 输出：应用内通知列表，以及之后的邮件/webhook 事件。
- 依赖：`CAP-004`、`CAP-005`、`CAP-006`、未来的认证/会话。
- 验收说明：
  - `FR-120` 在交换请求、接受/拒绝、反馈到期、模块评审结果以及收到反馈/评论时进行通知。
  - 除非之后的决策加入邮件，否则将默认通知渠道保持为应用内。
- 推迟项：邮件、GitHub issue/评论通知、webhook、agent 收件箱。

## 非功能性需求

| ID | 需求 | 验收说明 |
| --- | --- | --- |
| `NFR-001` | 隐私优先的数据最小化 | 任何发布关键路径都不得要求上传原始笔记、完整摘录、私有路径、凭据、私有嵌入向量或未脱敏的第三方个人数据。 |
| `NFR-002` | 设计上即 agent 可读 | 公共注册表和契约必须可通过结构化清单以及之后的 API 供 agent 使用。 |
| `NFR-003` | 可追溯的实现 | 每个页面、组件、API、模拟数据和测试都必须可追溯回稳定的 FR/NFR ID。 |
| `NFR-004` | 开源可审计 | 规则、schema、信任解释和隐私边界应可在仓库产物中查阅。 |
| `NFR-005` | 人类同意门 | 清单提交、联系和私下交换均需所有者明确掌控。 |
| `NFR-006` | 抗滥用 | 社交信号、反馈、提交和举报在公开发布前需要身份校验、唯一性约束、速率限制和审计轨迹。 |
| `NFR-007` | 可访问的公共网站 | 公开页面在前端实现开始后必须可键盘导航，并在桌面/移动端可读。 |

## 完整目标产品范围

设计范围是产品笔记和 UI 图像所表达的完整目标系统。设计阶段不应将产品拆分为 V0.1/V0.2/V0.3 发布。实现可以之后仍按依赖、风险和并行 agent 归属进行排序，但该排序不得缩小或重新定义目标产品。

完整目标系统包括：

- 公共发现注册表和全局站点外壳。
- 带有清单预览、隐私边界、来源统计、信任信号和交换入口的知识模块详情页。
- 带有本地生成/导入、隐私门、公开卡片预览、评审和私下交换包准备的清单提交工作流。
- 交换请求生命周期和公开交换记录，披露关系和结果而不暴露私有仓库内容。
- 结构化反馈、信任打分、贡献者档案、徽章和信任解释界面。
- 诸如收藏、认可、评论/反馈片段、举报和可审计性等社交信号。
- 用于本地清单创建、脱敏处理、验证、私有仓库打包和反馈提交的 Agent 技能 / MCP 工具界面。

- 用于交换、评审、反馈和社区事件的应用内通知界面。

实现顺序将在之后的产品规格、业务流程、领域模型、IA、页面规格、服务契约和模块工作流阶段决定。可能的依赖形态是：先有共享的清单/隐私/信任契约，再有页面/领域垂直切片，最后是集成和自动化。
