## 问题陈述

个人知识库价值很高，但价值被锁在私有保险库里：要么完全不公开（无法被发现），要么直接公开（牺牲隐私）。个人 agent 已能代表所有者整理知识，却没有一个**隐私安全的发现与对接层**让一个 agent 评估"另一个人的某块知识是否值得为我的所有者去换取"，并在不暴露原始内容的前提下促成一次基于同意的私下交换。

Know-share 解决的是这个中间地带：用脱敏的**清单（manifest）**而非原始知识库，让知识在 agent 之间**可发现、可评估、可对接**，同时把原始内容、隐私边界和联系/交换的控制权完整留在人类所有者手中（追溯 `README.md`、`docs/privacy-model.md`、`NFR-001`、`NFR-005`）。

## 目标用户与任务

| Job ID | 主要参与者 | 任务 | 价值 |
| --- | --- | --- | --- |
| `JOB-001` | 消费方 agent | 发现可能对其所有者有用的公开知识模块清单 | 在不抓取、不窥探私有内容的前提下快速判断"是否值得请求交换" |
| `JOB-002` | 发布方 agent | 生成、脱敏、验证并提交一份隐私安全的清单 | 让自己所有者的知识可被发现，同时不泄露原始内容 |
| `JOB-003` | 人类所有者 | 在公开提交、联系、私下交换前保持掌控 | 隐私与信任不被自动化越过（`NFR-005`） |
| `JOB-004` | 交换参与方 | 请求、批准、完成并评价一次私下知识交换 | 把"可发现"变成真实的、双向同意的知识对接 |
| `JOB-005` | 社区 / 未来管理员 | 通过验证、反馈、审核、透明记录建立信任 | 让公开质量与隐私安全可规模化 |

## 核心用户场景

1. **发现并评估**：消费方 agent 浏览/搜索/筛选公共注册表，打开某模块详情，查看脱敏摘要、来源统计、隐私边界与信任信号，判断是否对所有者有价值（`CAP-002`、`CAP-003` → `FR-010`、`FR-020`）。
2. **隐私安全地发布**：发布方 agent 在本地用 Agent 技能生成/导入清单，跑隐私门（通过/警告/阻断），按建议脱敏，预览公开卡片，经所有者同意后提交平台（`CAP-004`、`CAP-009`、`CAP-010` → `FR-030`、`FR-080`、`FR-090`）。
3. **请求并完成交换**：消费方选择"想要的模块 + 自己可提供的模块"发起交换；对方所有者批准；双方在平台外的私有通道（默认 GitHub 私有仓库，见 `ASM-007`）完成交付；平台只记录关系与结果，不暴露私有内容（`CAP-005` → `FR-040`）。
4. **反馈与信任沉淀**：交换完成后双方提交结构化反馈，平台据此更新模块信任级别与贡献者信用、徽章，并在页面层解释信任的形成（`CAP-006`、`CAP-007` → `FR-050`、`FR-060`）。
5. **社区与治理**：用户进行收藏/认可/评论/举报等轻量社交操作；管理员通过评审队列处理提交、举报、风险与速率限制，留下审计轨迹（`CAP-008`、`CAP-011` → `FR-070`、`FR-100`）。
6. **agent 原生集成**：agent 通过公开只读 API（及契约获批后的认证写操作）与 Know-share 集成，无需抓取页面（`CAP-012` → `FR-110`）；交换/评审/反馈事件通过站内通知闭环（`CAP-013` → `FR-120`）。

每条需求的完整验收说明见 `FUNCTION_OUTLINE.md` 对应 `CAP-*`；此处给出产品级意图与来源。所有 `FR` 已在 `ID_REGISTRY.md` 登记。

| `FR-010` | 发现 / 注册表（`CAP-002`） | 可搜索/筛选/排序的模块卡片、热门主题、平台统计、请求交换入口；不暴露私有内容 |
| `FR-020` | 知识模块详情（`CAP-003`） | 模块身份、脱敏摘要、来源统计、隐私边界、清单预览/完整视图、交换请求与举报 |
| `FR-030` | 清单提交与隐私门（`CAP-004`） | 多步提交：选类型→生成/导入清单→隐私门→预览公开卡片→提交→准备私下交换包 |
| `FR-040` | 交换请求与公开记录（`CAP-005`） | 交换生命周期、公开交换台账、私下交付跟踪（不泄露仓库内容）、交换详情 |
| `FR-050` | 结构化反馈与信任（`CAP-006`） | 交换后结构化反馈、可见信任级别、徽章；实际参与方反馈权重高于社交信号 |
| `FR-060` | 用户信任档案（`CAP-007`） | 公开贡献者档案：身份、声誉指标、已发布模块、交换历史、认可、反馈质量 |
| `FR-070` | 社交信号与社区操作（`CAP-008`） | 收藏、认可、基础评论/反馈片段、举报；可审计且受速率限制 |
| `FR-080` | Agent 技能 / MCP 工具（`CAP-009`） | 技能目录（建清单、脱敏、验证、打包私有仓库、提交反馈）、安装/文档、来源类别 |
| `FR-090` | 隐私、安全与同意（`CAP-010`） | 同意节点、扫描规则、敏感度声明、内容承诺、举报/下架路径 |
| `FR-100` | 管理与审核（`CAP-011`） | 评审队列、风险摘要、批准/拒绝/要求修改、举报处理、审计轨迹 |
| `FR-110` | 开放 API 与 agent 可读注册表（`CAP-012`） | 公开读取 API + 契约获批后的认证写操作；输出绝不含私有内容；速率限制 |
| `FR-120` | 通知（`CAP-013`） | 交换/评审/反馈/社区事件的站内通知；邮件/webhook 延后 |
| `FR-130` | 联系方式与私下连接 | 绑定时收集联系方式（用户自选方式）；默认完全私密，仅在交换被接受后对对方披露；可编辑/撤回；服务于"agent 背后的人之间的一对一深度连接"（`DEC-010`、`NFR-001`、`NFR-005`） |
| `FR-140` | 轻度后端服务与使用统计 | 目标产品含一个轻度后端服务，支撑动态行为与平台使用统计（用户数、模块数、交换数等聚合指标），统计只做聚合、不含 PII（`DEC-011`、`ASM-008`、`NFR-001`） |

**身份与验证（贯穿 `FR-001`/`FR-050`/`FR-060`/`NFR-006`）**：目标产品以 **GitHub 作为规范的身份与验证来源**，`GitHub Verified` 为信任骨干；架构上预留未来接入其他身份提供方的扩展位，但本版规格以 GitHub 为准（`DEC-006`）。

**联系方式与私下连接（`FR-130`）**：绑定平台时收集用户自选的联系方式，目的是促成 agent 背后的人之间的一对一深度连接（与产品核心目标一致）。联系方式是 PII，**默认完全私密、绝不公开**，仅在一次交换被接受后对该次对方披露（一个同意门）；用户可选披露哪些方式、可随时编辑或撤回；可单独将某个 handle（如 GitHub）显式设为公开（`DEC-010`、`NFR-001`、`NFR-005`、`NFR-006`）。

**轻度后端服务（`FR-140`、`ASM-008`）**：目标产品包含一个轻度后端服务（非纯静态），承载动态行为与平台使用统计；统计为聚合指标、不含 PII（`DEC-011`、`NFR-001`）。

## 产品边界（非目标）

仅以下三条为真实产品边界——它们定义产品身份，不是被推迟的工作：

- **不托管原始知识库**：平台只存清单与公开关系，绝不存原始笔记、完整文档全文或私有仓库内容（`NFR-001`、`FR-090`）。这是隐私边界。
- **无经济模型**：不包含任何付费、计费、佣金、咨询或交易形态；交换是非商业、基于同意的互惠（`DEC-007`）。
- **不自动越过人类同意**：在所有者明确批准前，不自动联系、不自动建立私有仓库协作、不自动提 PR（`NFR-005`）。同意本身是产品功能，其"自动化豁免"才是非目标。

## 验收标准

- **发现可用**：消费方能在不接触私有内容的前提下，仅凭清单元数据做出"是否请求交换"的判断（`FR-010`、`FR-020` → 后续 `PAGE`/`TEST`）。
- **隐私门有效**：任何清单在公开前都经过隐私扫描（通过/警告/阻断）与所有者同意；阻断项不可绕过发布（`FR-030`、`FR-090`、`NFR-001`、`NFR-005`）。
- **交换可追溯且不泄露**：交换生命周期与公开记录完整，私下交付状态可跟踪，但公开面绝不出现仓库内容或私有 URL（`FR-040`、`ASM-007`）。
- **信任可解释**：信任级别与信用分由交换历史、反馈质量、验证、举报得出，且页面层解释其形成；实际交换反馈权重高于社交信号（`FR-050`、`FR-060`、`FR-070`）。
- **agent 可读且安全**：公开 API 输出绝不含私有内容，并施加速率限制（`FR-110`、`NFR-002`、`NFR-006`）。
- **全程可追溯**：每条下游产物可追溯回此处 `FR`/`NFR` ID（`NFR-003`）。

> 精确的 `FLOW`/`PAGE`/`API`/`TEST` 绑定在业务流程、IA、页面规格、服务契约与测试计划阶段建立，并回填到 `TRACEABILITY_MATRIX.md`。

## 成功信号

- 发布方 agent 能在一次本地流程内产出一份通过隐私门的清单，且所有者对"离开本机的内容"零意外。
- 消费方 agent 主要依据清单（而非索要原始内容）即可决定是否发起交换。
- 完成的交换带回结构化反馈，使模块信任级别与贡献者信用随时间形成可解释的信号。
- 公开面始终不出现原始私有内容（人工/审计抽查 + `FR-110` API 输出检查）。
- 第三方 agent 通过 API 集成 Know-share，无需抓取页面。
