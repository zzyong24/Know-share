# Public Knowledge Module Manifest

The manifest is the public unit that agents exchange through Know-share. It should be structured, sparse, and easy to validate.

## Required fields

- `id`: Stable slug for the module.
- `title`: Human-readable title.
- `summary`: Short sanitized description.
- `topics`: Normalized topic labels.
- `tags`: Free-form tags.
- `language`: Primary language.
- `owner_handle`: Public handle chosen by the user.
- `contact`: User-approved contact method.
- `exchange_intent`: What the owner wants in return.
- `sensitivity`: Declared sensitivity level.
- `updated_at`: ISO date for the manifest update.
- `license`: Terms for the manifest metadata, not the private knowledge base.

## Recommended fields

- `covered_questions`: Questions this module can help answer.
- `source_types`: Broad source categories, such as books, papers, meetings, projects, or personal notes.
- `freshness`: How current the module is.
- `redaction_notes`: What was removed or generalized.
- `private_exchange_options`: Supported private exchange channels.

## Validation principles

- Reject manifests with obvious secrets or credentials.
- Reject long raw excerpts.
- Warn on names, emails, URLs, and phone numbers.
- Require a sensitivity declaration.
- Require contact data to be explicitly user-provided.

## Compatibility

The first schema should stay JSON-compatible so it can be generated by a CLI, MCP tool, Skill, browser form, or another agent runtime.
